# Runner trace / output contract

Status: design spec (docs only, no Go changes). Written for U0, consumed by U6b (executor
integration) and U6a (daemon protocol client). Grounds every claim in real bytes from the
current codebase, then defines the new format as the same kind of concrete bytes — this file
is the fixed target U6b builds against, not a description to reinterpret.

Session: `16-17-shellkit-upgrade`. Plan: `ccc-compound/plan.md`. Requirements:
`ccc-compound/requirements.md`.

**Scope guardrail (hard constraint #1, requirements.md:95-100):** the MCP DSL is frozen —
`###` step blocks, JSON config keys, `{{refs}}`, `$OUTPUT`, `entrypoint`, `timeout`, fan-out,
`jump`, `identity`, `continue_on_error` do not change. Everything below is either (a) provably
unchanged rendering, or (b) an additive/conditional rendering change that a script never
depending on trace text cannot observe.

## 0. What an agent sees, today (ground truth)

Two surfaces reach the agent: the **final tool-result text** (`formatResults`,
`internal/mcp/mcp.go:266-345`) and the **live ticker** (`ProgressSummary`,
`internal/mcp/events.go:187-218`, fired every 3s from `internal/mcp/mcp.go:142-159`). Both are
grounded below with real output, generated by calling the actual (unmodified) `formatResults`
against synthetic `StepResult` values in a throwaway test (`go test ./internal/mcp -run
TestScratchDump -v`, written, run, and deleted for this doc — not part of the diff).

### 0.1 Today's trace block, byte-for-byte

Input: a step with `Trace: [{ElapsedSec:0,Command:"cat /etc/os-release"},
{ElapsedSec:0,Command:"df -h /"}]`, `ShowTrace:true`, exit 0.

```
=== deploy-check [host-a] exit:0 ===
output: /tmp/shellkit-mcp-xxxx/deploy-check.out (3 lines)
  status=ok
command trace:
  +0s  cat /etc/os-release
  +0s  df -h /
preview:
  OS: Debian GNU/Linux 12 (bookworm)
  Disk: 57% used
  

```

Default (no `"trace": true`, not timed out) — the SAME step renders **without** the trace
block at all (`mcp.go:307`: `if len(r.Trace) > 0 && (r.TimedOut || r.ShowTrace)`), matching
`mcp-tests/09-command-trace.md` T09.1 ("trace is active but transparent — script output is
clean"):

```
=== deploy-check [host-a] exit:0 ===
output: /tmp/shellkit-mcp-xxxx/deploy-check.out (3 lines)
  status=ok
preview:
  OS: Debian GNU/Linux 12 (bookworm)
  Disk: 57% used
  

```

Timeout case (`TestFormatResults_WithTimeout`, and T09.2):

```
=== slow-step [host-a] exit:137 ===
TIMED OUT (after 30s)
command trace (timed out):
  +0s  echo starting
  +0s  apt-get update (12s)
  +12s  curl http://slow.example.com  ← timed out here
```

Section order, unconditionally (`mcp.go:269-341`):
`=== name [host] exit:N ===` → `TIMED OUT...` → `error: ...` → `output: ... (N lines)` →
`  k=v` (per output) → `stderr: ...` → `command trace...` → `preview:\n<indented>`.

### 0.2 Today's exact trace-line arithmetic (important — not what it looks like)

`prependTrace` (`mcp_exec.go:117-120`) fires bash's `DEBUG` trap **before** every simple
command, emitting `<marker> $SECONDS $LINENO $BASH_COMMAND`. So `TraceLine{ElapsedSec: t,
Command: cmd}` means "at time t we are about to run cmd" — cmd's own execution happens strictly
*after* that timestamp. `formatResults` (`mcp.go:322-333`) computes each line's shown duration
as `showTrace[i+1].ElapsedSec - showTrace[i].ElapsedSec` — the gap to the *next* trap firing.
Attribution is correct (the gap between two trap firings is exactly the preceding command's
run time) but resolution is whole seconds (bash `$SECONDS`), and **the last command in a
non-timeout trace never gets a duration** — there's no next line to diff against, and the
`else if r.TimedOut` branch only fires on timeout (`mcp.go:323-331`). A step that finishes in
under 1s renders every line as `+0s` with **no duration data at all** — the entire signal is
invisible, not just imprecise.

### 0.3 Today's ProgressSummary (the 3s ticker)

`events.go:199-217`:

```
executing (14s) step:deploy host:host-a
> git pull --ff-only
---
Cloning into 'app'...
remote: Enumerating objects: 412, done.
```

`> cmd` = latest "executing" event (DEBUG-trap marker, truncated 120 runes); `---` block =
last 8 stdout/stderr lines (`maxRecentLines`, `events.go:144`), each truncated to 200 runes.
No concept of "bootstrapping" — a step blocked on SSH connect or a slow remote just repeats
`executing (Ns)` with a stale or empty `>` line.

### 0.4 What the dashboard already tells us for free

`testdata/golden/*.expected` render a **different** surface (the human TUI dashboard, not
what the agent receives) but they prove something load-bearing: the dashboard's fractional
per-command durations (`(+0.26s)`, `(+3.46s)` in `pipeline.expected:6,10`) are **not** computed
from `TraceLine.ElapsedSec` (an int) — `internal/dashboard/log_dashboard_live.go:583-618`
computes waterfall timing from the event's own `ts` (RFC3339Nano) field, diffing consecutive
event timestamps. `ElapsedSec` is only read once, into `a.ExecutingElapsed`
(`log_dashboard_live.go:449`), for the coarse step-elapsed header. **This means the dashboard
is already forward-compatible with ns-precision trace data** — it ignores `ElapsedSec`'s
resolution for anything that matters and will render finer waterfalls for free once trace
events fire more often; nothing in `internal/dashboard` needs to change for U0/U6b.

## 1. New trace block (runner path)

### 1.1 Design

Two capture seams are structurally required (decision #7, plan.md:79): `ExecHandlers`
middleware sees only external commands (real argv, real per-command exit code, real
`dur_ns` measured at the seam — not inferred from a neighboring line); `CallHandler` sees
only builtins (no subprocess exit code — interp reports success/failure as a Go error, not
a wait-status). Per hard constraint #1 the merged trace stream renders **identically**
regardless of which seam produced a line — no `[ext]`/`[builtin]` tag, preserving the same
visual shape agents already parse. The seam split is an implementation reality for U3b, not
a rendering concern.

Proposed `TraceLine` shape (replaces `events.go`'s `ElapsedSec int`):

```go
type TraceLine struct {
    Seq        int    // wire frame sequence (ordering/dedup only, not rendered)
    ElapsedNS  int64  // ns since step start, captured at cmd_start
    DurationNS int64  // ns this command took, captured at cmd_end; 0 = unknown (builtin didn't report, or truncated mid-run)
    LineNo     int     // unchanged — source LINENO
    Command    string  // unchanged — argv-joined (external) or source text (builtin)
    ExitCode   *int    // external only; nil for builtins unless CallHandler surfaces a non-nil error
}
```

Rendering keeps the exact `mcp.go:332` shape — `  +<offset>  <command><suffix>` — swapping only
the number formatter (self-scaling, matches Go duration idiom):

```go
func formatDur(ns int64) string {
    switch d := time.Duration(ns); {
    case d < time.Microsecond:
        return fmt.Sprintf("%dns", ns)
    case d < time.Millisecond:
        return fmt.Sprintf("%.1fµs", float64(ns)/1e3)
    case d < time.Second:
        return fmt.Sprintf("%.1fms", float64(ns)/1e6)
    default:
        return fmt.Sprintf("%.3fs", float64(ns)/1e9)
    }
}
```

`exit:N` is appended inline **only when a command's own exit is non-zero** (mirrors the
existing "only render what's interesting" convention — `stderr:`/`error:` are conditional
too). This is a genuinely new capability: today's DEBUG-trap trace has no access to any
command's individual exit status (no `$?` capture in `prependTrace`), so a non-fatal failing
command in a `continue_on_error`/no-`set -e` script was invisible in the trace. The runner
path surfaces it because `ExecHandler` observes the real exit.

### 1.2 Worked example

Step body (3 commands, mixing builtin + external — `cd` and the `>>` redirect on `echo` are
builtins, `git pull` is external):

```
### deploy
{"ssh": "host-a", "interp": true, "trace": true}

cd /srv/app
git pull --ff-only
echo "deployed" >> $OUTPUT
```

**Today** (legacy path, same script, whole-second `$SECONDS` resolution — all three trap
firings land inside the same second because the whole step takes ~850ms):

```
command trace:
  +0s  cd /srv/app
  +0s  git pull --ff-only
  +0s  echo "deployed" >> $OUTPUT
```

No duration is shown anywhere — the signal a human/agent actually wants (git pull took most
of the 850ms) is not recoverable from this text.

**New** (runner path, same script, ns-precision `dur_ns` measured directly per command,
including the last one — closing the §0.2 gap):

```
command trace:
  +8µs      cd /srv/app                        (8µs)
  +20µs     git pull --ff-only                  (842.1ms)
  +842.2ms  echo "deployed" >> $OUTPUT           (4µs)
```

Same script, `git pull` failing (exit 1, script has no `set -e` so it continues — invisible
in the header's own exit code since the LAST command still exits 0):

```
command trace:
  +8µs      cd /srv/app                          (8µs)
  +20µs     git pull --ff-only        exit:1      (310.4ms)
  +310.5ms  echo "deployed" >> $OUTPUT             (4µs)
```

### 1.3 Do the caps and truncation survive?

**Yes, unchanged, by construction.** `mcp.go:314-321` (`cap := 20` / `50` when `ShowTrace`)
operates on `len(showTrace)`; `mcp.go:332`'s `mcpTruncate(t.Command, 500)` operates on the
`Command` string. Neither reads `ElapsedSec`/`ElapsedNS`. Swapping the timing field's type and
formatter touches zero lines of that logic. Section order is unchanged (§0.1) — the trace
block still sits between `stderr:` and `preview:`.

## 2. Mixed fan-out (host A runner, host B legacy)

`formatResults` already loops per-`StepResult` (`mcp.go:269`, `for _, r := range results`) —
each result renders its trace independently. A fan-out step where host A bootstraps the
runner successfully and host B falls back (e.g. `noexec` home dir) needs **no change** to
that loop; it needs each `StepResult` to carry its own route outcome. New field:

```go
type StepResult struct {
    // ...existing fields unchanged...
    RouteNote string // one line, e.g. "ran under legacy path: <reason>"; empty = runner path, no note
}
```

Rendered as a `note:` line right after the header — the same conditional-block convention as
`error:`/`stderr:` (new slot, least disruptive position; doesn't reorder or gate any existing
line).

Worked example — `check-nginx` fanned out to `["host-a", "host-b"]`, `host-a` warm on the
runner, `host-b` bootstrap fails (`noexec` cache dir) and silently falls back:

```
=== check-nginx [host-a] exit:0 ===
output: /tmp/shellkit-mcp-xxxx/check-nginx.host-a.out (2 lines)
command trace:
  +4µs    systemctl is-active nginx        (1.8ms)
  +1.9ms  echo "nginx: active" >> $OUTPUT   (4µs)
preview:
  nginx: active


=== check-nginx [host-b] exit:0 ===
note: runner bootstrap failed on host-b (noexec ~/.cache, /tmp also unavailable) — ran under legacy path
output: /tmp/shellkit-mcp-xxxx/check-nginx.host-b.out (2 lines)
command trace:
  +0s  systemctl is-active nginx
  +0s  echo "nginx: active" >> $OUTPUT
preview:
  nginx: active


=== check-nginx [local] exit:0 ===
output: /tmp/shellkit-mcp-xxxx/check-nginx.out (5 lines)
preview:
  === host-a ===
  nginx: active
  
  === host-b ===
  nginx: active
  

```

The two per-host trace shapes (ns-scaled vs. `+Ns` whole-second) coexist unmodified within the
same call, and the visual difference doubles as a diagnostic: an agent scanning the output can
tell which host ran which path from the trace granularity alone, without parsing `note:`. The
merged `[local]` block (`mcp_exec.go:689-709`, `mergeFanoutOutputs`) is untouched — it only
ever concatenated `Stdout`, never `Trace`.

## 3. ProgressSummary payload (live ticker during a runner step)

Decision #14 (plan.md:90): keep the 3s ticker cadence; its *content* becomes live trace state.
Design finding 7 / bootstrap E&R row (plan.md:247, "bootstrap failed: `<why>`, fell back")
requires a distinct phase so a cold-host push+exec-test reads as progress, not a stall.

Proposed shape — same `executing (Ns) step:X host:Y` prefix agents already parse
(`events.go:200-206`), with an added `phase:` token (additive; nothing existing breaks):

**Phase 1 — cold-host bootstrap** (no run frame sent yet; nothing to execute):

```
executing (1s) step:deploy host:host-b phase:bootstrap
> pushing shellkit-runner (linux/amd64, 1.4MB, sha:9f2a1c8e)
```

```
executing (2s) step:deploy host:host-b phase:bootstrap
> verifying runner (exec-test: shellkit-runner --version)
```

**Phase 2 — run**, once the runner is live. The `---` block now interleaves the **live trace
feed** (completed-command events, tagged `[trace]`) with the **stdout tail** (today's only
source) — this directly answers "live scrolling trace feed vs. stdout tail": both, tagged, so
a silent long-running command (no stdout) still produces visible progress from its own
completion event plus any commands that ran before it:

```
executing (5s) step:deploy host:host-b phase:run
> git pull --ff-only
---
[trace] cd /srv/app  (8µs)
Cloning into 'app'...
remote: Enumerating objects: 412, done.
```

If bootstrap fails outright, the next tick shows the fallback explicitly instead of silently
switching format (matches the `note:` line the final result will carry, §2):

```
executing (4s) step:deploy host:host-b phase:bootstrap-failed
> noexec ~/.cache, /tmp also unavailable — falling back to legacy path
```

## 5. Teaching-error copy

Follows the repo's `check:` deprecation precedent (`internal/mcp/mcp_parser.go:115-119`):

```go
return fmt.Errorf("step %q: the 'check' command was removed — run your verification commands through an ssh step instead, e.g. {\"ssh\": \"host\"}", s.Name)
```

Pattern: **names what's wrong** → **states what happened / what to do** → **gives a
copy-pasteable remedy**. Two categories apply here (E&R map, plan.md:236-238), rendered
differently:

### 5.1 Auto-route notices (router caught it statically — not an error, `note:` line, §2)

The router (U1) detects the construct before ever opening an SSH connection and silently
routes to the legacy path — "silently" to the *script*, but visibly to the agent via `note:`:

- Named-signal trap:
  `note: step uses "trap … INT/TERM" — interp supports only EXIT/ERR; ran under real bash (legacy path). Add "interp": false to silence this notice.`
- `kill`:
  `note: step uses "kill" — interp has no dispatch for kill (any form, not just %jobspec); ran under real bash (legacy path).`
- Job control (`jobs`/`fg`/`bg`/`disown`/`suspend`/`fc`/`newgrp`):
  `note: step uses "jobs" — interp has no job-control builtins; ran under real bash (legacy path).`
- Silent-divergence idiom screen (decision #16b — e.g. pipeline ending in a mutating builtin):
  `note: step ends a pipeline in "cd" — interp runs pipeline stages as goroutines, so a trailing cd/export/read would silently not affect later commands; ran under real bash (legacy path) to preserve real bash's subshell semantics.`

### 5.2 Teaching errors (router missed it — dynamic invocation, e.g. `eval`/`"$cmd"`)

No auto-rerun (E&R map, plan.md:237: "script may be non-idempotent"). Rendered as the
existing `error: %s` line (`mcp.go:285-287`), text follows the `check:` pattern exactly:

```
error: step failed under interp: unsupported builtin "kill" (interp supports only EXIT/ERR traps and has no job control) — this construct evaded the static gap-router, likely via eval/"$cmd"/alias; add "interp": false to force the legacy (real-bash) path. NOT auto-retried — the script may not be idempotent.
```

```
error: step failed under interp: invalid signal specification "INT" for trap (interp supports only EXIT/ERR) — likely a dynamically-built trap command; add "interp": false to force the legacy (real-bash) path. NOT auto-retried — the script may not be idempotent.
```

### 5.3 Protocol/bootstrap notices (E&R rows, plan.md:241,247 — same `note:` slot)

- Proto/role/version mismatch (stale runner binary): the handshake is rejected and the step falls
  back to legacy immediately (the daemon re-bootstraps the current runner on the next step, not
  mid-step):
  `note: runner protocol mismatch on host-b (runner proto:1, daemon proto:2) — ran under legacy path for this step.`
- Bootstrap `noexec`/read-only home:
  `note: runner bootstrap failed on host-b (noexec ~/.cache, /tmp also unavailable) — ran under legacy path for this step.`
- Darwin/arm64 unsigned binary (decision #18, plan.md:83):
  `note: runner exec-test failed on <host> (AMFI killed unsigned darwin/arm64 binary) — ran under legacy path for this step.`
